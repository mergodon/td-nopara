# td-flow

## What this is

The **td-flow** framework itself: a minimal, file-based, repo-portable working agreement for solo development. Same shape every project, conversational interface, structured docs in `.td/`, GitHub commits as memory. Nine slash commands (`/td-init`, `/td-clear`, `/td-close`, `/td-refresh`, `/td-mailbox`, `/td-health`, `/td-incident`, `/td-park`, `/td-snapshot`) — everything else is conversational.

## Who it's for

Solo developers working with Claude Code. Originally built for one user's portfolio; anyone can fork it for their own.

## Stack & choices

- Markdown-only — no compiled code, no npm package, no CLI binary.
- Bash for `install.sh` and the pre-commit hook.
- AWK for extracting the test command from `WORKWAY.md` § Local testing.
- Cross-project requests ride on GitHub Issues + `gh` CLI. No custom DB, no schema, no inbox service.
- **Tracker-free outbound.** Per-project `.td/PROJECT.md § Cross-repo` lists connected repos; `**From:** <project>` body marker on every cross-repo filing identifies the source. `/td-mailbox` does one bounded search across declared repos — no separate registry repo, no tracker Epic, no in-repo tracking infrastructure beyond the human-curated list and the body marker.
- Cloned to `~/projects/td-flow/` on each machine.
- Distributed as symlinks into `~/.claude/` — `commands/`, `skills/td-flow`, `td-templates`, and `td-flow-contract.md` (the contract every project `@import`s) — via `install.sh`.

## Out of scope (for now)

- Research / context7 deep integration in the rhythm — **decided against** 2026-05-20 (closed #7 as not-planned). The MCP is already wired up and ad-hoc use is already documented in CLAUDE.md. A formal "research phase" would add ceremony for every piece — anti the three-lines-beats-abstraction instinct.
- Subagents for implementation fan-out — **decided against** 2026-05-20 (closed #8 as not-planned). The Agent tool exists in the harness and research fan-out (Explore agents) is already documented; implementation fan-out can't be made fully safe (contract leaks, semantic conflicts past file-level isolation, partial-failure rollback). Coordination cost outweighs speedup for a solo dev where speed isn't the bottleneck. Revisit only with a real concrete case.
- npm package / CLI / Electron studio. Never.
- Claude Code plugin distribution — **evaluated 2026-05-22, ruled out.** The plugin system (marketplace + `/plugin install`) is the native way to ship slash commands + a skill, but it has no mechanism for a per-project standing contract: a plugin can't deliver a file each project `@import`s into its `CLAUDE.md`, and a plugin's own `CLAUDE.md` isn't loaded as project context. That import IS td-flow's core. A hybrid (plugin for the commands, `install.sh` for the contract) would mean two install mechanisms — less lean, not more. Clone + `install.sh` stays.
- Application-code test suite for the framework itself — there's no application code to test. Mechanical sanity checks (syntax, install idempotency, symlink integrity, AWK extractor) are automated as of v5.3 in `scripts/smoke.sh`, wired as the WORKWAY § Local testing Test command so the pre-commit hook runs them every commit. Semantic UAT (does `/td-snapshot` actually work end-to-end?) stays manual, against `mergodon/td-flow-test1`. Drift signals + "Before I commit a piece" ritual remain the self-validation.
